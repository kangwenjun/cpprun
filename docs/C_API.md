# extern "C"

## 提问
```
extern "C" 函数是外部接口吗？有没有统一的抽象名称？
```

---

## 结论

`extern "C"` **不等于**"外部接口"，它的本质是**链接约定（Linkage Convention）**，更准确地说是指定使用 **C ABI**。

有统一的抽象名称，但取决于你从哪个层级看。

---

## 逐层拆解

| 层级 | 名称 | 说明 |
|------|------|------|
| 语言层 | `extern "C"` | C++ 语法，告诉编译器"别做名称修饰" |
| 链接层 | **C Linkage（C 语言链接）** | ✅ 这是最直接、最统一的叫法 |
| ABI 层 | **C ABI** | ✅ 更底层的抽象，指调用约定、名称格式、栈帧规则 |
| 互操作层 | **FFI（Foreign Function Interface）** | ✅ 跨语言调用时的通用术语 |
| 设计层 | **Language Binding（语言绑定）** | ✅ 描述"两种语言之间的接口" |

---

## 为什么不是"外部接口"？

`extern "C"` 解决的问题很具体：

```cpp
// C++ 默认：名称被修饰（name mangling）
void foo(int);        // 编译后可能变成 _Z3fooi

// extern "C"：禁止修饰，用 C 的裸名称
extern "C" void foo(int);  // 编译后就是 foo
```

它**不定义接口语义**，只规定**名字长什么样、怎么调用**。

所以：
- ❌ "外部接口" → 太模糊，所有非 `static` 函数都有 external linkage
- ✅ **C Linkage / C ABI** → 精确描述它到底是什么

---

## 实际使用中最常见的叫法

| 场景 | 推荐叫法 |
|------|----------|
| 写文档 | **C ABI 接口** 或 **C 链接函数** |
| 跨语言调用（Python ctypes、Rust FFI 等） | **FFI 边界** |
| C++ 代码注释 | 直接写 `extern "C"` 即可，不需要额外命名 |

一句话总结：**`extern "C"` 的统一抽象名称是 C ABI（或 C Linkage），不是"外部接口"。**
